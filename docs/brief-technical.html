50pick is a Next.js 16 (App Router, RSC) product on Node.js 22, with PostgreSQL persistence on Railway, full TypeScript strict-mode, scrypt password hashing, HMAC-signed sessions, an HMAC-chained append-only audit log, and a 33-control regulator-aligned automated audit suite that runs against every push. Every state change is logged, every privileged action is server-side gated with defence-in-depth, every error path is bilingual (EN + SW). What follows is how each layer was chosen, what it protects, and how we'd answer a Stage-1 ISO 27001 auditor's first questions.

What this document does NOT contain

Implementation specifics that would help an attacker (rate-limit tuning, scrypt parameters, exact session expiry tolerances, source locations of the audit secret), proprietary betting-pool math (operator-margin algorithm, slippage handling), endpoint enumerations, test-script source. Those live in the private repository under standard need-to-know access controls. This brief is the surface — a reviewer can request specifics under NDA.

System architecture

How the pieces fit

Edge proxy (Next.js middleware)

Strict CSP · X-Frame DENY · nosniff · Referrer-Policy · Permissions-Policy · COOP · HSTS in prod · attaches request path to the per-request header so server components see the original URL.

↓

App Router (RSC) — Next.js 16

Server-rendered pages with streaming. Server Actions for every mutation (auth, bet, cashout, KYC, RG). Server Actions are CSRF-bound to the origin and the action ID; payloads are Zod-validated on the server.

↓

In-memory store (RAM)

Read & write hot path. Map-backed entities with strongly-typed mirrors of the Prisma schema.

⇄

Audit ring

Append-only, HMAC-chained. Each entry's hash includes the previous hash → tamper-evident.

↓ debounced 1.5 s

Persistence — single source of truth

DATABASE_URL set → PostgreSQL StoreSnapshot row (Railway). Unset → on-disk HMAC-signed JSON envelope. Never both. Boot path rehydrates from whichever is configured.

Why this shape (today)

Single-instance + in-memory + snapshot-to-Postgres lets us ship a soft launch with full ACID-like guarantees for a small bookmaker, without paying the complexity tax of a full multi-instance deploy. Every entity in the in-memory store has a one-to-one Prisma model (19 in the schema) — when traffic warrants horizontal scaling, the swap is method-by-method and tested incrementally. The architecture is not a long-term shape; it is a deliberate near-term shape with a documented migration path.

Where this goes next

Phase 4 of RAILWAY_DB_README.md describes the per-entity Prisma swap. After that: read replicas, then a queue-based async write path for the audit log so the chain becomes the slow tier and the request path stays sub-50ms even at peak.

Stack & rationale

Every choice, defended

Next.js 16 RSC

Server Components by default; Server Actions for every mutation. Eliminates an entire class of client-trust bugs (the client can't trigger a write without going through a server-validated action). Eliminates a separate API tier — the page is the API. Hot-path bundles ship under 80 KB.

TypeScript strict

The schema is the contract. Prisma generates client types from schema.prisma; the in-memory store types mirror them exactly. A field added in one place that's missing in another fails at compile.

Tailwind 3 + design tokens

Every colour, font, and spacing is a CSS variable defined once in the design kit. There are no hex values inline anywhere a regulator might inspect the CSS. Brand consistency by construction, not by review.

PostgreSQL + Prisma 6

Money handling demands transactional integrity. Prisma's parameterised queries make SQL-injection structurally impossible. Migrations are versioned in git and applied via prisma migrate deploy on every Railway deploy.

Zod validators

Every Server Action input is parsed through Zod before reaching the service layer. Bad shapes return a clean { ok: false, code } result, never a 500.

scrypt password hashing

Per-user 16-byte salt. Resistant to GPU brute-force at the parameters used. Common-password blacklist applied before hashing so weak inputs never reach the database.

HMAC-SHA-256 sessions

HttpOnly + SameSite=Lax + Secure-in-prod cookies. Tamper-evident — flip any byte and verification rejects. 7-day absolute cap with a 24-hour idle timeout.

Token-bucket rate limit

Per (key, action) buckets. Both per-phone and per-IP keys for register/login (defence against credential-stuffing and mass-account abuse). Refills slow enough to stop attackers, fast enough to not annoy real users.

Playwright test harness

136 controls across 6 suites. Real Chromium against a real dev server. Every push runs them locally before commit.

Audit-log integrity (the regulator question)

Every state change emits one entry: category, action, actorId, targetType, targetId, payload, ip, userAgent, createdAt, plus prevHash and entryHash. entryHash is HMAC-SHA-256 over the canonical JSON of the row including prevHash. The chain is verifiable in O(n): walk forward, recompute each hash, compare against the stored one. Any modified or removed entry breaks at the affected index — the verifier reports the first break, which is what a forensic reviewer needs.

Data model

19 entities, drawn at the right altitude

The entity boundaries match the regulator's mental model: identity (User, Session, OTP, Device), KYC, money (Wallet, Transaction), betting (Market, MatchEvent, Window, Pool, BetBundle, Bet), integrity (AntiFraudFlag, MatchIntegrityCheck), compliance (ResponsibleGambling, Notification, AuditLog), and partners (AffiliateAgent, ProviderHealth). A regulator asks for a bet's audit trail; we walk the chain in three steps.

Why these specific lines

  • User is identity + role. KYC, RG, Wallet are separate entities with FKs back. Avoids the temptation to bloat User and lose composability.
  • Wallet is one-to-one with User. Holds balance + pending + hold separately so an in-flight withdrawal cannot be double-spent.
  • Transaction is the immutable ledger entry. Status transitions only forward (PENDING → CONFIRMED / FAILED). Reversal is a NEW transaction.
  • Bet belongs to a Market via a BetBundle (multi-market accumulator). Stakes are integer TZS — no floating point.
  • Market has resolution-source URL + two-officer sign-off fields baked in. A market without these can't be created.
  • AuditLog is append-only. prevHash + entryHash form a Merkle-style chain. Verifiable in O(n).

Why these are NOT shown here

  • Field-level types and enums.
  • Index choices.
  • The pool-share calculation algorithm.
  • The exact AML threshold.
  • Audit-chain HMAC key location.

These live in the private repository. Engineering hires get day-one access; partners get specifics under NDA.

Deployment & operations

How we ship · how we scale

Today (soft-launch shape)

Single Railway service running Next.js. Single Railway-managed PostgreSQL. git push triggers a redeploy in 2–3 minutes; prisma migrate deploy runs automatically on every build, so a schema change is one commit + one push. Daily Postgres backups on Railway with 7-day retention; weekly external archive to S3 Glacier Deep Archive for the 5-year regulator retention window.

Next (production hosting)

  • Per-entity Prisma swap (store.ts → 10 Prisma services). 60–90 min of focused work per entity slice; runbook in RAILWAY_DB_README.md Phase 4.
  • Read replica for analytics + admin dashboards.
  • Connection pooler (Railway's pgbouncer endpoint already wired; one env-var swap).
  • Multi-instance + sticky sessions behind a load balancer. The session cookie is signed not stored, so any instance can verify any session — no shared cache needed.
  • Async audit-log writer (queue + worker). The chain becomes the slow tier; the request path stays sub-50 ms even at peak.
  • External SMS (Selcom or Beem) and mobile-money aggregator wired through the existing provider abstractions — flag flips, no code change.

Observability runbook

/api/health exposes uptime, user count, audit-ring size, market counts, and SMS provider success rate — Railway's built-in health check polls it. Application logs go to Railway's log retention (7 days on Hobby, 30 on Pro). Each request log line carries a request-id; an audit entry that points to a request-id can be joined against the application log for full forensic context.

What a partner sees on day-one engagement

A handover packet that's already drafted: this brief + the operator brief + the design kit + the Prisma schema + the regulator-audit test suite. A senior engineer can read those and ship a feature in a week without asking us a single onboarding question.
